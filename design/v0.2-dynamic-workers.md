# Dyanamic Workers

## Background

The original version of lelouch had the user manually create new worktrees and
register them. This approach results in some gaps:

- Difficulty dynamically resizing worker count relative to demand.
- Lack of sandboxing of the agent. This has not been an issue thus far, but can result in:
  - Accidental deletion of files in the parent filesystem.
  - Unfettered access to local credentials.

## Proposal

The new version (v0.2) will instead _actively_ orchestrate these worktress,
introducing the concept of _workspaces_. The approach will be as follows:

1. Instead of specifying worktrees manually, the user will include a max_worker_count in the
   repository configuration:

```toml
[[repositories]]
name = "my-project"
path = "~/git/my-project"
executor = "antigravity"
max_worker_count = "2"
```

upon launching lelouch, lelouch will:

1. Create max_worker_count worktrees.
2. Read the work database for the total number of issues that are ready.
3. For each issue that is ready:
   1. launch a docker container with the desired executor running inside of it.
      1. the docker container will have the worktree mounted inside of it.

## Design Details

## Worktree location

lelouch will create a directory `~/.lelouch/worktrees` and create worktrees inside it.

The worktrees are prefixed with the repository name, and suffixed with a unique identifier.

## Persistent worktrees

Although one could create the worktrees _dynamically_, this can result in
significant latencies if the repository is large (e.g. a monorepo).

Therefore, the default functionality is to create and persisted the worktrees.

## Version Control Interface

Rather than couple to a specific version control interface, there will be an
interface for version control, similar to the executors today.

## Dockerfile location

If a Dockerfile exists in the project root, that will be used.

This can be overrided with a `dockerfile` field in the repository configuration.

## Passing agent credentials

To run agents, credentials are typically required.

Every executor (e.g. `gemini`) will contain a default directory which it will
_also_ be mounted into the container. For example for gemini on linux, this is
`$HOME/.gemini`.
